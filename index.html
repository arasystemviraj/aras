1.0 The Single-Board Vulnerability

Traditional electronic control units (ECUs) and DIY single-board microcontrollers (e.g., standard ESP32 or Raspberry Pi implementations) suffer from critical architectural vulnerabilities when deployed in life-safety applications. This vulnerability is formally known as "Thread Starvation" or "Blocking I/O Delay."

When a single processor attempts to simultaneously poll high-frequency acoustic sensors, render I2C graphics to an OLED, handle WiFi/Bluetooth radio overhead, and actuate physical brakes, the CPU scheduler is forced to queue tasks. A mere 200-millisecond delay in processing—caused by something as trivial as a SPIFFS flash memory write-cycle or a dropped TCP/IP packet—creates a catastrophic blind spot.

The Mathematics of Latency

At 80 km/h (22.2 m/s):
A 200ms processing lag = 4.44 meters traveled entirely blind.
In a panic-braking scenario, a 4.4m margin of error results in a fatal kinetic transfer.

2.0 The Dual-MCU Solution Paradigm

TwinKey abandons the single-board paradigm entirely. By physically separating the computational logic across a Dual-MCU architecture, tasks are strictly siloed. This is the core basis of Patent Application 202521120497.

  • MCU-A (The Actuator Node)

    Operates on a continuous, uninterrupted cyclic executive loop. It performs exactly two functions: firing the 5-10 ultrasonic arrays and energizing the 5V brake relays. It has zero internet access, hosts no UI, and writes no files. Its execution time is mathematically guaranteed.

  • MCU-B (The Telemetry Node)

    Receives serialized, non-blocking state payloads from MCU-A via a high-speed hardware UART line (115200 baud). It handles all "heavy" operations: formatting the CSV, conducting atomic writes to SPIFFS flash storage (~1GB/month), driving the OLED displays, and serving the port 80 web dashboard.

Structured Abstract

Background. Low-cost two-wheeler assistive braking stacks commonly fail to isolate life-safety decisions from user-interface, networking, and persistent-storage workloads, creating scheduler-induced latency spikes at the exact moment deterministic actuation is required.[R1]

Objective. This dossier formalizes TwinKey ARAS as a modular dual-MCU rider-assist platform intended to preserve deterministic relay actuation while expanding sensing context through environmental telemetry and mobile computer vision.[N1]

Method. A hardware-isolated actuator node polls distributed ultrasonic sensors and executes relay logic, while a secondary logger node handles rendering, storage, and communications. A tethered smart device provides contextual classification, OCR, and gyroscope-assisted false-positive suppression.[R3]

Contribution. The architecture claims novelty in physically decoupled brake actuation, adaptive hazard envelopes, gyro-cancelled lean suppression, and a fusion pipeline combining low-cost acoustic sensing with quantified mobile inference.[R5]

Section 03

System Theory and Hazard Model

The proposed system is based on a simple thesis: in a low-cost embedded rider-assist stack, the dominant safety risk is not merely sensing error but computational interference between sensing, rendering, storage, and actuation. TwinKey therefore treats latency isolation as the primary architectural safety requirement.[R1][R2]

Figure 1. Isolation model showing that only the actuator node resides on the critical actuation path, while visualization, storage, and contextual AI remain downstream or advisory.[N3]

Hazard Taxonomy

H-01 Scheduler Contention

Occurs when blocking storage, display, or radio tasks preempt collision-response logic. Mitigation is physical compute segregation.

H-02 Acoustic Ambiguity

Occurs when range sensing cannot distinguish object semantics. Mitigation is mobile vision enrichment and contextual policy gating.

H-03 Lean-Induced False Positives

Occurs during aggressive turns as lateral sensors read asphalt reflections. Mitigation is gyroscope-informed lateral suppression.

H-04 Partial System Failure

Occurs when logging/UI subsystems reboot or disconnect. Mitigation is fail-operational Node-A behavior and manual hydraulic preservation.

Thermal & Power Management

Designed for the extreme thermal environments of the Indian subcontinent (specifically tested against Rahata/Maharashtra summer profiles exceeding 45°C ambient), the system avoids linear voltage regulators (like the AMS1117) for main power distribution, which bleed excess voltage as heat.

Instead, the 12V motorcycle battery is stepped down using high-efficiency LM2596 DC-DC Buck Converters. This ensures stable 5V rails for the ultrasonic arrays and relay coils, while a secondary ultra-low-dropout regulator provides the 3.3V logic level required by the ESP8266 nodes.

Section 05

Interconnects, Timing, and Signal Flow

The interconnect scheme is intentionally minimal: every channel must justify itself by improving determinism, observability, or serviceability. No communication path is allowed to pull user-interface or storage workloads into the actuation loop.

UART Backbone

Node-A emits state vectors to Node-B without waiting for acknowledgements, ensuring logger or display failures cannot stall hazard evaluation.

Relay Control Plane

Brake assist decisions remain local to the actuator node, avoiding queue-backed commands or web-triggered actuation semantics.

Diagnostic Mirror

The logger receives a mirrored state history for reconstruction, allowing replay and forensic analysis with zero dependency on the critical path.

Algorithmic Drag Adjustments

The code snippet illustrates the dynamic threshold logic. A static braking distance of 2 meters might be sufficient on dry asphalt at noon, but is lethally inadequate during a monsoon night.

The `calculateDragCoefficient()` function acts as a localized weather engine. If the DHT11 reads >80% humidity (indicating a wet road with lower friction), the system artificially inflates the `criticalDistance` multiplier by 30%, causing the system to intervene earlier.

Edge AI Computer Vision

Hardware sensors are physically blind to context. An ultrasonic sensor cannot differentiate between a slowing truck and a stationary road barrier. ARAS utilizes Edge AI to solve this.

YOLOv8 + MOBILENET SSD

TensorFlow Lite Quantization

To achieve Level-2 autonomy characteristics without requiring massive onboard GPUs, TwinKey ARAS pairs via Bluetooth to a custom Android application. Utilizing the smartphone's camera array mounted on the dashboard, the system runs object detection models locally on the phone's NPU (Neural Processing Unit).

Running standard YOLOv8 on a mobile device is computationally prohibitive. The model is converted to TensorFlow Lite and undergoes INT8 Quantization. This converts 32-bit floating-point weights into 8-bit integers, reducing the model size by 4x and increasing inference speed significantly, at a negligible ~1-2% drop in mAP (Mean Average Precision).

Section 10

Rider Interface and Human Factors Engineering

The rider should not need to interpret a paragraph while moving at speed. Human factors design therefore separates urgent tactile signaling from glanceable visual context and from post-ride analytics.[R4]

Figure 2. Conceptual rider notification layout emphasizing lateral tactile mapping and low cognitive overhead.

Principles

  • Urgency should be communicated through pulse pattern and side placement, not dense text.
  • Visual detail should scale down as vehicle speed or maneuver intensity scales up.
  • Assistive braking must remain additive, predictable, and non-mysterious to the rider.
  • False positives damage trust faster than limited capability; suppression quality is therefore central to adoption.

Validation & Safety Assurance

A life-safety system must be proven. This section details the False-Positive testing and thermal resilience of the TwinKey architecture.

False-Positive Mitigation

A major issue with ultrasonic sensors on motorcycles is lean-angle triggering. When banking hard into a corner, the side-mounted sensors will reflect off the asphalt, causing a false-positive collision alert. TwinKey solves this by utilizing the Android smartphone's IMU Gyroscope.

Demonstrating the reduction in false braking events when Gyro-Cancellation logic is applied.

🛡️ Fail-Safe, Not Fail-Deadly

If MCU-B (Logger/UI) fails, loses power, or enters a boot loop, MCU-A (Actuator) continues to operate independently, ensuring the brake relays and haptics remain fully functional. If MCU-A fails, an inline mechanical bypass ensures the rider's standard manual hydraulic braking remains 100% unaffected. The system can only *add* braking force, never prevent it.

Acoustic Cross-Talk Prevention

With up to 10 HC-SR04 sensors emitting 40kHz pulses, the echo of Sensor 1 can easily be read by Sensor 2, creating phantom obstacles. The RTOS loop resolves this by multiplexing the triggers using a Time-Division Multiple Access (TDMA) approach, guaranteeing acoustic silence before the next pulse.

Cost Model, Commercial Positioning, and Scalability

The commercial relevance of TwinKey lies in narrowing the gulf between hobbyist prototypes and inaccessible automotive ADAS platforms. The design intentionally targets a cost-performance region that is realistic for retrofit experimentation.

Unit Economics Summary

  • Prototype BOM remains within student-builder reach.
  • Incremental cost is dominated by enclosure quality, harnessing, and ruggedization rather than compute.
  • Software complexity scales more slowly than hardware repeatability requirements.

Positioning Thesis

TwinKey is positioned not as a replacement for premium automotive ADAS, but as a low-cost assistive safety layer for two-wheelers where affordability, modularity, and explainability matter as much as raw sensor sophistication.

Decoupled Dual-MCU Actuation Interlock

An electronic safety system for two-wheeled vehicles comprising: a primary microcontroller unit (MCU) strictly and exclusively dedicated to physical sensor polling and hardware relay actuation; a secondary microcontroller unit securely linked via serialized communication exclusively dedicated to user interface generation, wireless telecommunications, and non-volatile data storage. Wherein said architecture completely isolates life-safety brake actuation threads from latency induced by network or file-writing operations, mathematically guaranteeing actuation response times of less than 100 milliseconds.

[02]

Contextual Hazard Evaluation Matrix

The system as claimed in claim 1, further comprising a programmatic methodology for dynamic threshold adjustment wherein base acoustic proximity readings are continuously multiplied against environmental drag coefficients derived from onboard Light Dependent Resistors (LDR) and Digital Humidity/Temperature (DHT) sensors; wherein the system autonomously expands its physical braking envelope during conditions determined to represent low-visibility or reduced road surface friction (e.g., >80% relative humidity).

[03]

Hybrid Sensor-Vision Data Fusion Pipeline

A method for collision mitigation utilizing the hardware specified in claim 1, comprising the bluetooth transmission of localized sensor state vectors to a tethered smart device; wherein the smart device utilizes onboard optical hardware and quantized YOLO object detection models to visually classify physical obstacles. The system cross-references visual classification against acoustic proximity to dictate context-aware levels of haptic feedback intensity to the vehicle operator (e.g., ignoring static guardrails while highlighting crossing pedestrians).

[04]

Gyro-Canceled Acoustic Validation

A computational mechanism for the suppression of false-positive collision alerts on tilting vehicles, comprising an algorithmic fusion of multiaxial gyroscope data and lateral ultrasonic echoes. Wherein the system dynamically ignores acoustic proximity alerts on the lateral flank corresponding to the direction of the vehicle's lean angle, recognizing said echoes as road-surface reflections rather than foreign physical obstacles.
