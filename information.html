Problem: Thread Starvation

Single-MCU safety stacks (ESP32, Raspberry Pi) must share one scheduler between collision logic, display rendering, WiFi, and flash writes. A 200 ms SPIFFS write or dropped TCP packet creates a 4.44 m blind spot at 80 km/h — enough to cause a fatal collision.

Solution: Dual-MCU Isolation

TwinKey physically separates the actuation path from all non-critical tasks onto two ESP8266 modules linked by unidirectional UART. The actuator node (Node-A) only polls sensors and drives relays. The logger node (Node-B) handles everything else.

Structured Abstract

Background. Low-cost rider assist stacks fail to isolate life-safety decisions from UI, networking, and storage workloads, creating scheduler-induced latency spikes at the exact moment deterministic actuation is required.

Method. A hardware-isolated actuator node polls distributed ultrasonic sensors and executes relay logic; a secondary logger node handles rendering, storage, and communications. A tethered smartphone provides contextual classification, OCR, and gyroscope-assisted false-positive suppression.

Contribution. Physically decoupled brake actuation, adaptive hazard envelopes, gyro-cancelled lean suppression, and acoustic-vision fusion on low-cost commodity hardware.

Firmware Logic

// Dynamic threshold based on DHT11 + LDR readings
float calculateDragCoefficient(int humidity, int light) {
    float drag = 1.0;
    if (humidity > 80) drag += 0.3; // Wet road: expand braking envelope 30%
    if (light < 20)    drag += 0.2; // Low visibility: expand 20%
    return drag;
}

void loop() {
    unsigned long now = millis();
    if (now - previousPing >= PING_INTERVAL) {
        previousPing = now;
        for (int i = 0; i < NUM_SENSORS; i++)
            distances[i] = getDistance(sensorMap[i]); // TDMA — no cross-talk

        float drag = calculateDragCoefficient(readDHT(), readLDR());
        int   critDist = BASE_THRESHOLD * drag;
        bool  hazard   = false;

        for (int i = 0; i < NUM_SENSORS; i++) {
            if (distances[i] > 0 && distances[i] < critDist) {
                hazard = true;
                triggerHaptics(i); // Directional side alert
                break;
            }
        }
        digitalWrite(BRAKE_RELAY_PIN, hazard ? HIGH : LOW);
        transmitStateToLogger(distances, hazard); // Fire-and-forget UART
    }
}

System Theory & Hazard Model

The primary safety risk in a low-cost rider-assist stack is not sensing error — it's computational interference between sensing, rendering, storage, and actuation. TwinKey treats latency isolation as the primary architectural safety requirement.

H-01 Scheduler Contention

Blocking storage, display, or radio tasks preempt collision-response logic. Mitigation: physical compute segregation.

H-02 Acoustic Ambiguity

Range sensing cannot distinguish object semantics. Mitigation: mobile vision enrichment + contextual policy gating.

H-03 Lean-Induced False Positives

Lateral sensors read asphalt reflections during aggressive cornering. Mitigation: gyroscope-informed lateral suppression.

H-04 Partial System Failure

Logger / UI reboots or disconnects. Mitigation: fail-operational Node-A + manual hydraulic bypass.

Edge AI & Sensor Fusion

A tethered Android device runs quantized TensorFlow Lite object detection on its NPU. INT8 quantization reduces the model 4× and improves inference speed with ~1–2% mAP loss. Acoustic proximity and visual classification are cross-referenced before any brake intervention.

Rider Interface Principles

  • Urgency communicated through haptic pulse pattern + side placement, not text.
  • Visual detail scales down as speed or maneuver intensity increases.
  • Assistive braking must be additive, predictable, and non-mysterious.
  • False positives destroy trust faster than limited capability — suppression quality is central.

Acoustic Cross-Talk Prevention

Up to 10 HC-SR04 sensors firing 40 kHz pulses risk mutual echo contamination. Node-A multiplexes triggers via TDMA — guaranteeing acoustic silence on all inactive sensors before the next pulse fires.

Patent Claims — Application No. 202521120497

[01] Decoupled Dual-MCU Actuation Interlock

An electronic safety system for two-wheeled vehicles comprising: a primary MCU strictly and exclusively dedicated to physical sensor polling and hardware relay actuation; a secondary MCU linked via serialized communication exclusively dedicated to UI generation, wireless telecommunications, and non-volatile data storage. Wherein said architecture completely isolates life-safety brake actuation threads from latency induced by network or file-writing operations, mathematically guaranteeing actuation response times of less than 100 milliseconds.

[02] Contextual Hazard Evaluation Matrix

The system as in claim 1, further comprising dynamic threshold adjustment wherein base acoustic proximity readings are continuously multiplied against environmental drag coefficients derived from onboard LDR and DHT sensors; wherein the system autonomously expands its physical braking envelope during conditions representing low-visibility or reduced road surface friction (e.g., >80% relative humidity).

[03] Hybrid Sensor-Vision Data Fusion Pipeline

A method for collision mitigation utilizing the hardware in claim 1, comprising Bluetooth transmission of localized sensor state vectors to a tethered smart device; wherein the smart device uses onboard optical hardware and quantized YOLO object detection models to visually classify physical obstacles. The system cross-references visual classification against acoustic proximity to dictate context-aware haptic feedback intensity (e.g., ignoring static guardrails while highlighting crossing pedestrians).

[04] Gyro-Canceled Acoustic Validation

A mechanism for suppression of false-positive collision alerts on tilting vehicles, comprising algorithmic fusion of multiaxial gyroscope data and lateral ultrasonic echoes. Wherein the system dynamically ignores acoustic proximity alerts on the lateral flank corresponding to the vehicle's lean direction, recognizing said echoes as road-surface reflections rather than physical obstacles.

Novelty Separation

Novel core: Physical separation of actuation from non-critical tasks; context-aware threshold expansion; hybrid acoustic-vision fusion; gyroscope-assisted lateral suppression on a tilting platform.

Non-novel substrate: Generic use of ultrasonic sensors, common MCU boards, smartphone pairing, or dashboard displays — these are enabling implementation detail, not the inventive step.
